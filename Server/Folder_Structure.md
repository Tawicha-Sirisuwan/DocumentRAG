# โครงสร้างโฟลเดอร์ของ Server (Python FastAPI)

การจัดโครงสร้างนี้ช่วยให้โค้ดเป็นระเบียบ ดูแลรักษาง่าย และรองรับการขยายโปรเจกต์ในอนาคต

```text
Server/
│
├── app/                     # โฟลเดอร์หลักสำหรับเก็บโค้ดทั้งหมดของแอปพลิเคชัน
│   ├── main.py              # ไฟล์จุดเริ่มต้นของโปรเจกต์ (ตั้งค่า FastAPI และเชื่อมต่อ Router)
│   │
│   ├── api/                 # โฟลเดอร์สำหรับแยก Routes (Endpoint ของ API)
│   │   ├── chat.py          # รวม API เกี่ยวกับการแชท (เช่น POST /api/chat)
│   │   └── document.py      # รวม API เกี่ยวกับเอกสาร (เช่น POST /api/upload)
│   │
│   ├── core/                # โฟลเดอร์เก็บการตั้งค่าหลัก (Configuration)
│   │   └── config.py        # โค้ดดึงค่าจากไฟล์ .env หรือตั้งค่าตัวแปรระบบ
│   │
│   ├── models/              # โฟลเดอร์สำหรับ Pydantic Models
│   │   └── schemas.py       # กำหนดหน้าตาของ Request/Response (เช่น คลาส ChatRequest)
│   │
│   ├── services/            # โฟลเดอร์เก็บ Logic หรือการทำงานหลักของระบบ
│   │   ├── rag_service.py   # โลจิกดึงข้อมูลและให้ AI ตอบคำถาม (Retrieval & Generation)
│   │   └── vector_db.py     # โลจิกเชื่อมต่อกับ Vector Database (เช่น ChromaDB)
│   │
│   └── utils/               # โฟลเดอร์สำหรับฟังก์ชันช่วยเหลือ (Helper functions)
│       └── file_handler.py  # เช่น ฟังก์ชันอ่านไฟล์ PDF หรือจัดการไฟล์อื่นๆ
│
├── venv/                    # (ถูกสร้างอัตโนมัติ) โฟลเดอร์ของ Virtual Environment
├── requirements.txt         # ไฟล์บันทึกรายชื่อไลบรารีที่โปรเจกต์ต้องใช้
├── .env                     # ไฟล์เก็บรหัสผ่าน หรือค่า Configuration ลับ (ไม่ควรเอาขึ้น Git)
└── .gitignore               # ไฟล์บอก Git ว่าไม่ต้องสนใจไฟล์ไหนบ้าง (เช่น .env, โฟลเดอร์ venv/)
```

## หน้าที่ของแต่ละส่วน

1.  **`main.py`**: จะเอาไว้แค่ Start Server และตั้งค่าเริ่มต้นของแอปพลิเคชัน รวมถึง Import ส่วน Routes อื่นๆ เข้ามา เพื่อให้ไฟล์หลักไม่รกและอ่านง่าย
2.  **`api`**: ทำหน้าที่รับ Request จากผู้ใช้ (เช่น หน้าเว็บส่งข้อมูลมา) ตรวจสอบความถูกต้องเบื้องต้น ก่อนส่งข้อมูลต่อไปให้ฝั่ง `services` ทำงานต่อ
3.  **`services`**: เป็น "สมอง" ของระบบ โลจิกที่ซับซ้อนทั้งหมดจะอยู่ที่นี่ เช่น การเชื่อมต่อฐานข้อมูล การทำ RAG การเรียกใช้ AI (ข้อดีคือถ้าในอนาคตต้องการเปลี่ยน API Framework ส่วนของ logic นี้จะยังสามารถนำไปใช้ต่อได้โดยแทบไม่ต้องแก้)
4.  **`models`**: ใช้กำหนดรูปแบบของข้อมูล (Data Structures) ช่วยตรวจสอบ (Validate) ว่าข้อมูลที่ส่งเข้ามาทาง API นั้นครบถ้วนและถูกต้องตามรูปแบบที่เราต้องการหรือไม่ ก่อนที่จะนำไปประมวลผลต่อ
5.  **`core`**: เป็นจุดรวมการตั้งค่าทั้งหมด เช่น การดึงค่า API Keys หรือตั้งค่าคงที่ต่างๆ ของระบบ
6.  **`utils`**: แหล่งรวมฟังก์ชันเล็กๆ น้อยๆ ที่สามารถเรียกใช้ซ้ำๆ ได้จากหลายๆ ที่ เช่น ฟังก์ชันแปลงวันที่ หรืออ่านไฟล์
